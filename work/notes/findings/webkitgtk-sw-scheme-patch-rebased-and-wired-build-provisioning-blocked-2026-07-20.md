The build-and-measure record for the deferred half of the WebKitGTK SW-scheme fork spike (ADR-0016 decision 6). This host (AMD Ryzen 7 PRO 6850U, 16 cores, 30 GB RAM, 146 GB free, cmake 3.31 + ninja 1.12 + ruby 3.3 + gcc 14 + zig 0.16 + xvfb-run, full host spec in the task sidecar `host-spec.txt`) has the RAW resources ADR-0016 said the dev box lacked. What it does NOT have is WebKitGTK's set of build-dependency `-dev` packages, and installing them needs root, which the autonomous agent that did this work has no passwordless access to. So the patch was fully REBASED and the whole Zig side was ACTIVATED + proven, but the actual multi-hour C++ compile + the live-SW proof are gated on ONE privileged `apt-get install` a human must run. The task sidecar carries a `build-patched-webkitgtk.sh` that performs exactly that install + the build + the proof in one documented step.

## DONE here (environment-independent + measured)

- **Patch REBASED onto the exact tagged tree (2.52.3).** The sidecar draft from locate-and-draft was located against WebKit `main` with approximate line numbers; it was re-anchored line-for-line against the extracted 2.52.3 source and captured as `work/tasks/ready/spike-webkitgtk-sw-scheme-patch-build-and-measure/webkitgtk-2.52.3-sw-scheme-capable.patch` (the true, applies-cleanly diff for this tree).
- **Rebase friction MEASURED (the fork-cost datum this task owns).** See the section below. Short version: the policy hunks were STABLE (both `ServiceWorkerContainer` predicates + the `LegacySchemeRegistry` pair + the GLib API glue landed verbatim on the CORS/secure anchors), and the rebase surfaced only MECHANICAL deltas the `main`-anchored draft could not have known: the public header is a generated TEMPLATE (`WebKitSecurityManager.h.in`, not `.h`), and the UI->web IPC sync spans two more files than the draft enumerated (`WebProcess.h` + `WebProcessCreationParameters.{h,serialization.in}` for the late-web-process replay the footprint note flagged). 11 files touched, ~130 added / ~2 changed lines. No policy logic changed on rebase; only the file SET grew by the mechanical mirror the draft under-counted.
- **Seam wiring ACTIVATED (guarded), core gate GREEN.** `src/system_webview_renderer.zig::declareSchemeSecurity` now calls `webkit_security_manager_register_uri_scheme_as_service_worker_capable(mgr, scheme)` for the `service_worker_capable` trait, but ONLY under the compile-time `shell_options.sw_patch` flag (`-Dsw-patch`, default OFF). The default build never references the patched-only symbol, so `zig build shell` still links against stock WebKitGTK on the bare CI runner. A matching `isSchemeServiceWorkerCapable` query mirrors `isSchemeSecure` (also `sw_patch`-guarded). `zig build test` stays green; `chrome_conformance` stays green (the new WebKit call is in the allowed backend file).
- **Shipped `ipfs://` trait FLIPPED to service-worker-capable.** `ipfs_scheme.secure_origin_traits` gains `service_worker_capable = true` (a decision locate-and-draft deferred to this task; see its Decisions block and this note's below). On an unpatched backend the trait is a declared-but-unhonoured no-op (ADR-0016 d.5), so the flip changes nothing on stock WebKitGTK / CI / the existing `ipfs-secure-origin-test` leg. The live SW leg uses this SHIPPED constant, so it proves exactly what release wires.
- **Off-core-gate live-SW leg LANDED.** `zig build ipfs-sw-hosting-test` (mode `ipfs-sw`, `src/shell.zig::ipfsSwHostingTest`) serves an `ipfs://` page + its `sw.js` + a fetch sentinel through the interception hook under Xvfb, registers the SW, waits until it controls the page, fetches the sentinel, and asserts the SW's `fetch` handler's marker reaches the page (distinguishing "SW never registered" from "SW registered but fetch never ran"). The step is registered in `build.zig` ONLY under `-Dsw-patch` (it needs the patched lib), so a default `zig build` / the bare CI runner never sees it. It COMPILES on the default (unpatched) build (the patched symbol is compiled out); it can only RUN once the patched lib exists.

## BLOCKED here (needs a privileged dependency install + the multi-hour compile)

- **Patched WebKitGTK build + build-time measurement.** A `cmake` configure against the extracted 2.52.3 tree fails at the first missing dep (`LibGcrypt`), and a probe shows a large missing `-dev` set: `libgcrypt20-dev`, `libgstreamer1.0-dev` + `-plugins-base1.0-dev`, `libseccomp-dev`, `libwpe-1.0-dev` + `libwpebackend-fdo-1.0-dev`, `libmanette-0.2-dev`, `libgbm-dev`, `libwoff-dev`, `libavif-dev`, `libjxl-dev`, `liblcms2-dev`, `libenchant-2-dev`, `libnotify-dev`, `libhyphen-dev`, `libopenjp2-7-dev`, `libsecret-1-dev`, `gperf` (and a few more). `sudo` on this host is password-gated; the autonomous agent has no password, so it cannot install them.
- **The live SW-hosting proof (SW `fetch` runs on a secure `ipfs://` page).** Gated on the same build: the `ipfs-sw-hosting-test` leg needs the patched `libwebkitgtk` at `-Dsw-webkit-prefix`.

### To unblock (one documented step)

Run `work/tasks/ready/spike-webkitgtk-sw-scheme-patch-build-and-measure/build-patched-webkitgtk.sh` on a host where you can `sudo` (the reference host, with a password). It installs the deps (the only privileged line), fetches+verifies+extracts the source, applies the rebased patch, configures a spike build (heavy optional features off), compiles + installs into its own prefix (never overwriting the system lib), records the build TIME to `build-time.txt`, and runs `zig build ipfs-sw-hosting-test -Dsw-patch -Dsw-webkit-prefix=...` to PASS the live SW proof. After it prints the build time + the PASS line, the last two acceptance criteria are closed; append the build time + the observed SW-fetch PASS to this note.

## Rebase-friction measurement (fork-cost datum)

| Layer (locate-and-draft footprint) | On rebase to 2.52.3 |
|-------------------------------------|----------------------|
| WebCore predicate (2 clauses) | **verbatim** — both `protocolIsInHTTPFamily()` gates in `ServiceWorkerContainer::addRegistration` present + unchanged; the added `&&` clause dropped in cleanly. |
| WebCore registry pair | **verbatim idiom** — mirrored the 2.52.3 CORS pair exactly (`isMainThread()` guard, `makeNeverDestroyedSchemeSet`, `contains<StringViewHashTranslator>`), placed next to `CORSEnabledSchemes()`. |
| GLib API glue | **verbatim** — the `SecurityPolicy` enum value, the two switch cases, and the two public functions mirror `_as_cors_enabled` exactly. ONE correction the `main`-anchored draft missed: the public header is `WebKitSecurityManager.h.in` (a build-time template), not `.h`. |
| UI<->web IPC sync | **grew by 2 files** the draft folded into "~20 lines": the draft named `WebProcessPool.{h,cpp}` + `WebProcess.{cpp,messages.in}`; the working rebase ALSO needs `WebProcess.h` (the receiver decl) and `WebProcessCreationParameters.{h,serialization.in}` (the late-web-process replay list). This is the exact "creation-parameters replay" hunk the footprint note flagged as most likely to grow — confirmed. |

**Read:** the POLICY of the patch is rebase-stable (every load-bearing hunk re-anchors on the adjacent CORS/secure sibling that WebKit maintains in lockstep). The only friction is MECHANICAL and BOUNDED: keep the trait wired through every place the sibling `_as_cors_enabled`/`_as_secure` surface is wired (one enum, two switch cases, one registry pair, one pool method + member, one IPC message, one web-process receiver, one creation-param field + its serialization, plus the two predicate clauses). A per-release rebase is a mechanical re-mirror, not a design redo. This is the "small, localized, mirrors existing code" lower bound the locate-and-draft footprint predicted, now confirmed against a real tagged tree. The remaining fork-cost datum (actual build TIME) is what the build script measures once deps are provisioned.

## Decisions

- **`ipfs_scheme.secure_origin_traits` now carries `service_worker_capable = true`.** locate-and-draft deliberately left it OFF (it could not build the patched backend, so flipping it would only have driven the stubbed no-op and muddied the stock-WebKitGTK secure-origin proof) and handed THIS task the choice. Flipped ON here so the SHIPPED `ipfs://` constant is what the live SW leg exercises and what a `WezigRenderer` reproduces. Safe because the trait is a no-op on any backend that does not honour it (ADR-0016 d.5), so stock WebKitGTK, the bare CI runner, and the existing `ipfs-secure-origin-test` leg are unaffected. Touches: `ipfs_scheme.zig` (the constant + its "traits are exactly secure+CORS" core-gate test, updated to assert SW-capable), the new `ipfs-sw-hosting-test` leg. Alternative considered — keep it off and pass an ad-hoc SW-capable trait only in the live leg — rejected: it would prove a capability the shipped constant does not declare, so the proof would not match release.
- **`-Dsw-patch` (bool, default off) + `-Dsw-webkit-prefix=<path>` gate the patched build.** A NEW build-option pair, needed because the patched-only symbol must be compiled out on the default/CI build (which links stock WebKitGTK). `-Dsw-patch` also gates REGISTRATION of the `ipfs-sw-hosting-test` step, so a default `zig build` cannot even try to run it against an unpatched lib. Coherence check vs CONTEXT.md/ADRs: no existing flag or glossary term means this; it sits at the right layer (a build/link-time selector on the disposable webview backend, ADR-0005, never reaching the chrome); it does not duplicate an existing concept (the seam already declares the trait uniformly — this flag only decides whether the WebKitGTK backend HONOURS it, exactly ADR-0016 d.5's "whether a backend can honour it varies"). It is spike-only and NOT wired into `.goreleaser.yaml` / the default `zig build shell`, per the "patched build is a spike, not release-wired" criterion. Touches: `build.zig`, `shell_options`, `system_webview_renderer.zig`, `shell_main.zig`. Alternative considered — auto-detect the patched symbol via a config probe — rejected as more machinery than a spike needs and less explicit about "this is the patched build".

## Why the residual split (the environment reason, again)

locate-and-draft split the spike because the dev box could not build WebKitGTK at all. On this provisioned host the resources exist, so EVERYTHING except the privileged dep install + the compile is now done and green. The residual gate is narrower than ADR-0016 anticipated: not "no capable machine" but "the capable machine needs one `sudo apt-get install` a human must run". The build script reduces that to a single command; once run, the two build-gated acceptance criteria close and the keep-fork-vs-defer-vs-shim ratification data (patch size + upstream draft from locate-and-draft; rebase friction measured here; build time from the script) is complete.

## UPDATE 2026-07-20 — the patched build RAN; the fork's real result measured

The privileged install + build was run by a human on the reference host. The two build-gated criteria are now RESOLVED, and the resolution REVISES the plan (see ADR-0018).

### Build: SUCCEEDED
- Patched WebKitGTK 2.52.3 built in **57m03s (`build_seconds: 3423`) on 16 cores** (host spec in `host-spec.txt`), into the private prefix. Build-time fork-cost datum.
- Build deps exceeded the paper list: configure also needed `-DENABLE_SPEECH_SYNTHESIS=OFF` (flite), `-DUSE_LIBBACKTRACE=OFF` (libbacktrace), the `unifdef` build tool, and `-DENABLE_WEBDRIVER=OFF` — surfacing one at a time across repeated configures. Only knowable by building; the script is updated with the full working set.
- Patched symbols ARE in the built lib: `nm -D libwebkitgtk-6.0.so* | grep -c service_worker_capable` = 6. The fork works at the binary level.

### Live SW proof: register() UNBLOCKED, but SW `fetch` NOT observed (the finding)
`zig build ipfs-sw-hosting-test -Dsw-patch -Dsw-webkit-prefix=<prefix>` against the patched lib ran to a verdict (it did NOT hang): **`ServiceWorkerFetchNotObserved`**.
- Proven: `serviceWorker.register()` on the `ipfs://` origin is NO LONGER REJECTED on the patched build — the exact block stock WebKitGTK enforces (and this patch targets) is removed. The fork's primary claim holds.
- Not achieved: a full SW `fetch`-interception round-trip. The test registers the SW, `clients.claim()`s, fetches a sentinel, expects the SW `fetch` marker back; the marker never arrived within 60s.
- Not diagnosed further: WebKitGTK does not surface the SW-internal reason without content-process debugging. Distinguishing a SECOND WebKit gate on SW-fetch-interception for a custom scheme vs a content-process propagation gap vs an SW-on-custom-scheme client-control semantic is deep WebKit-internal work — the "fighting an engine we don't control" cost the ratification weighs.

### Ratification: DEFER to `WezigRenderer` (ADR-0018)
Measured: the minimal patch is **necessary but not sufficient** (unblocks register, not fetch); full SW-hosting on a custom scheme exceeds it. Weighed against a ~1hr per-release fork build + maintenance tail for an INCOMPLETE capability, the decision (ADR-0018) is: do NOT carry the fork; make `WezigRenderer` the home for `ipfs://` SW hosting; keep the webview backend with this feature ACCEPTED as degraded/unavailable. The rebased patch + `-Dsw-patch`/`-Dsw-webkit-prefix` stay reproducible EVIDENCE (spike-only, never release-wired), not an adopted fork. The upstream proposal may still be filed; wezig does not wait on it.

**Acceptance-criteria status (final):** (1) builds + time recorded ✅; (2) SW `fetch` observed end-to-end ❌ — register() unblocked but fetch not observed; per ADR-0018 this is the DECISION-DRIVING finding, not a defect to force green; (3) fork cost measured (build 57m + rebase friction) ✅; (4) seam contract unchanged, `chrome_conformance` green ✅; (5) spike artifact, not release-wired ✅. Four of five met; criterion 2 resolved as "measured negative → defer to `WezigRenderer`".
