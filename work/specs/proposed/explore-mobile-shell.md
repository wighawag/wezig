> **This is an EXPLORATION-scoped spec.** Its deliverable is CONFIDENCE + a de-risked, sliced build plan for running wezig on iOS and Android ‚Äî NOT a shipped mobile app. It proves that the TWO pinned seams from `explore-webview-shell` (the `Renderer` content seam and the `Toolkit` chrome-host seam, ADR-0006) carry a real, interactive page on a `WKWebView` (iOS) and an Android `WebView` backend WITHOUT the chrome changing, and it spikes the parts most likely to be hard or wrong: the Zig‚ÜíiOS and Zig‚ÜíAndroid toolchain (build, link, package, sign), the opaque `ViewHandle` crossing a NON-GTK toolkit‚Üîbackend boundary (ADR-0007 flagged cross-toolkit embedding as an unproven spike, not a drop-in), and whether the two web3-load-bearing hooks (script-message bridge + custom-scheme interception) survive on both mobile webviews. Each spike stays on the NARROWEST real case (one page, one bridge round-trip, one scheme request, one device/emulator). The actual mobile browser BUILD (packaging pipeline, app lifecycle, full chrome, store delivery) is a follow-on build spec this exploration de-risks and plans.

## Resolved decisions (the exploration's fixed points)

The six questions that gated tasking are DECIDED below (accepted 2026-07-17). They are the fixed points the spike executes against; a spike may still surface a FINDING that revises one (that is the exploration's job), but tasking no longer waits on them. Load-bearing ones land in the mobile ADR (story 12).

1. **Target matrix and floor ‚Äî DECIDED.** **iOS 16.0** floor, arm64 devices, spike on the **iOS Simulator** (arm64 on Apple Silicon, x86_64 on Intel). **Android API 26 (8.0)** floor, arm64 + x86_64, spike on an **x86_64 emulator**. Rationale: iOS 16 covers the overwhelming majority of active devices with modern `WKWebView`/`WKURLSchemeHandler` semantics and no legacy workarounds; API 26 is where the Android System WebView became an auto-updating Chromium component, so `shouldInterceptRequest`/`evaluateJavascript` track a modern engine. The long tail below these floors is deliberately dropped ‚Äî early wezig users are not on it.
2. **Zig‚Üímobile toolchain ‚Äî DECIDED: Zig static library + a thin native shell project per platform** (NOT driving the whole app from `build.zig`). iOS: `zig build-lib` ‚Üí `aarch64-ios` / `aarch64-ios-simulator`, linked into a minimal Xcode/SwiftPM target that owns the `WKWebView`, Info.plist, entitlements, and signing; Zig‚ÜîSwift over a C-ABI header (Zig `export fn`). Android: `zig build-lib` ‚Üí `aarch64-linux-android` / `x86_64-linux-android` (Zig's first-class Android libc targets), packaged by a minimal Gradle/NDK project with a small JNI shim driving `android.webkit.WebView`. The OS toolchains own signing/packaging/entitlements; Zig owns the portable core and all seam logic. (The windowing sub-question is answered by Q4: mobile gets no `createWindow`/`run`/`quit`.)
3. **`ViewHandle` sufficiency ‚Äî DECIDED: keep it opaque (`*anyopaque`) and prove it carries a mobile native view.** iOS backend returns the `WKWebView`'s `UIView*`; Android backend returns a JNI global-ref to the `WebView`. Embedding is expected to hold because the mobile chrome-host is native code on the same side of the FFI that created the view (it downcasts the opaque handle exactly as `gtk_toolkit.zig` does). The ONE risk the spike must confirm: on Android the handle is a JNI reference (lifetime + thread-affinity), not a raw pointer ‚Äî if the opaque contract cannot cleanly carry it, that is the highest-value FINDING and feeds back to ADR-0006/0007 (likely a typed-handle / handle-with-vtable refinement).
4. **Chrome-host / lifecycle model ‚Äî DECIDED: mobile needs a distinct chrome-host contract, expressed as a SPLIT of `Toolkit`, not a brand-new seam.** Split `Toolkit`'s concerns along the axis that actually differs desktop‚Üîmobile: a **chrome-surface** half (`embedView`, `setUrlText`, `setBackEnabled`/`setForwardEnabled`, `setChromeCallback` ‚Äî the widgets + intents both platforms implement) and a **host/loop** half (`createWindow`/`present`/`run`/`quit` ‚Äî desktop-only; on mobile the OS IS the host and owns the run loop, the chrome living inside a `UIViewController`/`Activity`). `chrome.zig` stays unchanged; the desktop/mobile difference becomes "which host drives the loop," not two unrelated seams. This is a real (small) refinement of the pinned `Toolkit` interface, done now on the mobile spike's evidence ‚Äî the same discipline ADR-0006 used for deferring input/scroll until a second backend forces it.
5. **The two web3 hooks ‚Äî DECIDED: confirm parity; expect it to hold with TWO documented semantic gaps to encode at the seam.** iOS maps cleanly (bridge ‚Üí `WKUserContentController` + `WKScriptMessageHandler` + `evaluateJavaScript`; scheme ‚Üí `WKURLSchemeHandler`), with the lifecycle constraint that custom schemes must be registered on the `WKWebViewConfiguration` BEFORE the webview is created (the seam should surface this ordering). Android has the sharper gaps the spike must record: `addJavascriptInterface` callbacks and `shouldInterceptRequest` run on NON-UI (binder) threads (native must marshal back to the UI thread), and custom-scheme security-origin/`isSecure` traits differ (Android treats custom schemes as opaque/insecure by default, affecting secure-context/service-worker behaviour on served content ‚Äî directly relevant to `ipfs://`). These gaps ARE the "scheme security traits at the seam" item ADR-0007 deferred; proving the hooks on mobile forces that decision, so `explore-web3-capabilities` inherits a cross-platform-validated hook contract.
6. **Mobile verification ‚Äî DECIDED: mirror the `shell-*` pattern one tier down, in dedicated legs kept OUT of `zig build test`.** iOS: boot the iOS Simulator headlessly via `xcrun simctl`, install the spike app, drive it, assert navigate + a `finished` lifecycle event + a non-blank snapshot (simulator screenshot) ‚Äî runs on macOS runners ONLY. Android: boot a headless x86_64 emulator (`-no-window`) via `avdmanager`/`emulator`, install via `adb`, run an instrumented test asserting the same three things. The core `zig build test` gate stays device-free (only the `Fake*` seam contracts run there). Cost realism: iOS proofs require macOS CI (‚âà10√ó Linux minutes), so the mobile proof legs run **on-demand / nightly**, not on every PR.

> These decisions resolved the spec's launch open-questions; `needsAnswers` is cleared. The spike is expected to CONFIRM them and may feed back findings (esp. Q3 Android JNI-handle sufficiency and Q4 the `Toolkit` split) to ADR-0006/0007.

## Problem Statement

wezig is being built as "a browser as a browser should be", and a browser people actually use has to be where they are ‚Äî which today means iOS and Android as much as the desktop. The webview-shell track (ADR-0005/0007) is how wezig becomes usable and differentiated long before the native renderer matures, and its whole architectural bet is TWO pinned, swappable seams: the `Renderer` content backend and the `Toolkit` chrome host, with a chrome that talks only to those seams and reaches neither `webkit_` nor `gtk_` (machine-checked by `src/chrome_conformance.zig`). That design PROMISES mobile is a backend swap ‚Äî a `WKWebView`/Android `WebView` `Renderer` and a UIKit/Android `Toolkit`, with the chrome and the web3 hooks unchanged. But nothing in the repo has proven that promise: the opaque `ViewHandle` is, in practice, a GtkWidget both current backends agree on (ADR-0007 explicitly calls cross-toolkit view embedding a "foreign-embedding spike, not a drop-in"); input/scroll forwarding is deferred at the `Renderer` seam; and Zig-into-an-`.ipa`/`.aab` (cross-compile, link, JNI/Obj-C interop, signing, packaging) is entirely unproven here. The mobile OSes also own the window and run loop, which the desktop `Toolkit` currently owns itself. Before committing to a mobile BUILD, we need CONFIDENCE that the seams really do carry mobile, the toolchain really works, and we know HOW the mobile browser grows ‚Äî not a mobile app written on guesses.

## Solution

Explore, don't build the mobile app. Spike each risky part on the narrowest real case, behind the EXISTING seams wherever they hold, and feed any insufficiency back as a seam finding rather than working around it:

- **Toolchain spike (per platform).** Prove the `wezig` Zig library cross-compiles and links into a minimal native shell on each platform, and that the shell launches: iOS (static lib into an Xcode target, a `WKWebView`, code-signed to a simulator), Android (NDK build into an APK, JNI to an `android.webkit.WebView`, run on an emulator). Deliverable is a launching skeleton + a pinned, written toolchain path ‚Äî not a real app.
- **Seam-attachment spike (per platform).** Implement a mobile `Renderer` backend (iOS `WKWebView`, Android `WebView`) satisfying the pinned `Renderer` interface, drive ONE real page through it (navigate + one lifecycle event observed + the view non-blank), and host that view through a mobile `Toolkit`/chrome-host ‚Äî proving the opaque `ViewHandle` and the two seams carry mobile, OR pinning exactly where they don't.
- **Web3-hook spike (per platform).** Round-trip ONE bridge message both ways (`window.wezig.ping`) and serve ONE custom-scheme request (`wezig-test://hello`) through the mobile backend, mirroring the desktop `shell-bridge-test`/`shell-scheme-test` proofs, so the web3 capabilities are confirmed backend-agnostic across desktop AND mobile.
- **Verification strategy.** Decide and prove a simulator/emulator-based, scriptable proof leg (kept OUT of the display-free `zig build test` gate, exactly like the Xvfb `shell-*` steps), so the mobile seam can be regression-tested without a physical device.
- **Evaluate + decide the open questions** (target floor, toolchain pinning, `ViewHandle` sufficiency, chrome-host/lifecycle mapping, hook parity, verification), capture the load-bearing ones in an ADR, and emit a de-risked, SLICED mobile build plan (which follow-on build spec grows packaging / app lifecycle / full chrome / store delivery, in what order, to what bar).

Keep the existing seams (`Renderer` ADR-0005/0006, `Toolkit` ADR-0006) as the structure the mobile work extends; the chrome-binding-free discipline (`chrome_conformance.zig`) and the desktop `shell-*` proofs are the prior art the mobile proofs mirror.

## Implementation Decisions

Decisions made at launch to seed tasking (trimmed into tasks/ADRs at tasking-time):

- **Extend the existing seams; do not fork them.** Mobile `Renderer` backends implement the pinned `Renderer` VTable; the mobile chrome host implements (or minimally extends) the `Toolkit` seam. The chrome (`src/chrome.zig`) stays untouched and binding-free; the mobile backends are the ONLY code that touches platform webview/UI APIs, exactly as `system_webview_renderer.zig`/`gtk_toolkit.zig` are the only WebKitGTK/GTK touchers on desktop. Any need to change a pinned interface is a FINDING fed back to ADR-0006/0007, not a silent divergence.
- **`ViewHandle` stays opaque; prove it carries a mobile native view.** The iOS backend returns a `WKWebView`/`UIView` as the opaque handle; the Android backend a `WebView`/`View`. The spike either confirms the mobile `Toolkit` can embed it unchanged, or pins the exact contract the seam is missing.
- **Input/scroll forwarding remains deferred (as on desktop).** Mobile webviews self-handle input/scroll like the desktop system webview, so the seam's deferred input methods are not blocked by this spec; the exploration only NOTES that a future `WezigRenderer`-on-mobile would need them (consistent with ADR-0006's deferral), not adds them speculatively.
- **Verification mirrors the `shell-*` pattern, one tier down.** Dedicated build/CI legs boot a simulator/emulator, drive one navigation, and assert a lifecycle event + a non-blank snapshot ‚Äî kept OUT of `zig build test`, exactly like the Xvfb `shell-test`/`shell-bridge-test`/`shell-scheme-test` steps. The mobile bridge/scheme proofs mirror `shell-bridge-test`/`shell-scheme-test` message-for-message.
- **A thin native shell project per platform is expected** (an Xcode target for iOS, a Gradle/NDK project for Android) hosting the Zig library, rather than driving the whole app from `build.zig`. The exact division of labour between `build.zig` cross-targets and the native project is a Q2 spike output.

## Testing Decisions

- **Seam contracts stay in `zig build test`, headless and device-free.** Any mobile-agnostic seam logic added is exercised through the existing `FakeRenderer`/`FakeToolkit` fakes in the display-free gate; no simulator/emulator is required for the core gate. The REAL mobile proofs (a page renders on `WKWebView`/Android `WebView`) live in the dedicated per-platform legs, mirroring how the WebKitGTK proof lives only in `shell-test` under Xvfb.
- **Test EXTERNAL behaviour, per the desktop prior art.** Assert what the desktop proofs assert ‚Äî a page loads (a `.finished` lifecycle event reached a subscriber), the view is non-blank (a snapshot), the bridge round-trips both ways, the custom scheme serves a native body that renders ‚Äî not platform-API internals. Prior art to copy: `renderer.zig`'s seam-contract tests (fakes, in-gate) and `shell_main.zig`'s `smoke`/`bridge`/`scheme` modes (real backend, under the wrapper).
- **One device surface per platform for the spike** (one iOS simulator, one Android emulator). Physical-device matrices, performance, and store validation are out of scope for the exploration.

## Out of Scope (this is exploration)

- **A shipped mobile browser** ‚Äî packaging pipeline hardening, full mobile chrome (tabs, gestures, settings), app lifecycle/state restoration, permissions UX, notifications, store submission/signing for release. The exploration PROVES the seams + toolchain and PLANS the build; BUILDING the mobile app is a follow-on spec this de-risks.
- **The native renderer on mobile** (`WezigRenderer` running on iOS/Android) ‚Äî this spec uses the SYSTEM webview backends only; the native-renderer-on-mobile question is downstream of both this spec and `explore-native-renderer`.
- **The web3 capabilities themselves** ‚Äî this spec only confirms the two HOOKS carry on mobile webviews; the EIP-1193 provider and `ipfs://` resolution are `explore-web3-capabilities` and its follow-on builds.
- **The desktop shell and its seams** (`explore-webview-shell`, ADR-0005/0006/0007) ‚Äî inherited as fixed points, not re-opened here (except to receive any seam-insufficiency finding this exploration surfaces).
- **Physical-device testing, cross-device performance, and the full OS-version support matrix** beyond the one floor chosen for the spike (Q1).

## Further Notes

- **Why a fourth exploration spec rather than folding mobile into `explore-webview-shell`.** `browser.md` was deliberately re-scoped (2026-07-14) to avoid one spec mixing a committed slice with gated direction, per "task a spec atomically or split it". Mobile has exactly the shape of the existing three explorations ‚Äî open questions, unproven approach, needs a spike + a build plan, not a build ‚Äî so it is authored as its own confidence-tier spec, ordered behind the seam it depends on. This keeps each spec atomically taskable.
- **What the seam design already buys mobile (the reason to be optimistic).** The two web3 hooks map near 1:1 to platform primitives (iOS `WKUserContentController` + `WKURLSchemeHandler`; Android `addJavascriptInterface`/`evaluateJavascript` + `WebViewClient.shouldInterceptRequest`), and the chrome is already machine-proven binding-free ‚Äî so the differentiators and the chrome are positioned to be a backend swap. The exploration exists to convert "positioned to be" into "proven to be".
- **The sharpest risk to watch.** ADR-0007 already flagged cross-toolkit view embedding (a non-GTK chrome hosting a foreign view) as an unproven spike; mobile makes that a first-class, must-answer case (Q3/story 6) rather than a desktop edge case. If the opaque `ViewHandle` proves insufficient, that finding reshapes the pinned `Renderer`/`Toolkit` interfaces and is the most valuable single output of this exploration.
- **Landscape note (mid-2026 framing).** System-webview-backed cross-platform browsers are well-trodden (every non-Chromium mobile "browser" on iOS is a `WKWebView` host by App Store policy); wezig's differentiation on mobile is the SAME native Ethereum-provider + `ipfs://` story attached at the seam, not the webview hosting itself. That is why proving the two hooks on mobile (stories 8‚Äì9) is load-bearing for the whole mobile value proposition.
