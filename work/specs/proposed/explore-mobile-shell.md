## Open questions

These gate tasking this spec (`needsAnswers: true`). Exploration exists to answer them; answering them is a primary OUTPUT, not a precondition.

1. **Target matrix and floor.** Which OS versions, architectures, and devices are the supported floor for the exploration and the eventual build — iOS (min version, arm64 device + simulator on arm64/x86_64) and Android (min API level, arm64 + x86_64 emulator)? This sets what "runs on mobile" is measured against and what the CI/dev spike must provision. Deciding the floor IS a primary output.
2. **Zig→mobile toolchain, pinned.** How does the Zig library cross-compile and link into each platform's app package: iOS (static library into an Xcode/`.ipa` target, code signing, Objective-C/Swift interop for `WKWebView`), Android (the NDK path into an `.aab`/`.apk`, JNI to drive `android.webkit.WebView`)? Which build driver orchestrates it (raw `build.zig` cross-targets + a thin native shell project, vs a platform toolchain invoking Zig)? The exploration SPIKES the load-bearing path on each platform rather than deciding on paper. Does a `Toolkit`/windowing analogue even make sense on mobile, where the OS owns the window/activity lifecycle (see Q4)?
3. **`ViewHandle` sufficiency across a non-GTK toolkit↔backend boundary.** Today `ViewHandle` is `*anyopaque` that "a GtkWidget both backends agree on" (ADR-0006/0007). On iOS it is a `WKWebView`/`UIView`; on Android an `android.webkit.WebView`/`View`. Confirm BY ATTACHING that the opaque handle carries these and that a mobile `Toolkit` can embed a mobile `Renderer`'s view — OR find the seam needs a richer contract for cross-native embedding, and feed that back to ADR-0006/0007 as a seam finding. This is the single most likely place the pinned interface proves insufficient.
4. **Chrome-host / lifecycle model on mobile.** The desktop `Toolkit` owns the window, the toolbar widgets, AND the main event loop (`run`/`quit`). On mobile the OS owns the window, the activity/scene lifecycle, and the run loop; the chrome is hosted INSIDE a native `UIViewController`/`Activity`. Decide how the `Toolkit` seam maps here: does `GtkToolkit`'s shape (create-window / run-loop / URL-bar widgets) translate to a mobile toolkit implementation, or does mobile need a distinct chrome-host contract (and if so, is that a NEW seam or an extension of `Toolkit`)? Resolve on the narrowest real case (one screen, a URL field, back/forward), not a full mobile chrome.
5. **The two web3 hooks on both mobile webviews.** Confirm the script-message bridge (`injectUserScript`/`setScriptMessageHandler`/`evaluateScript`) and the custom-scheme interception (`registerScheme`) map to `WKUserContentController` + `WKURLSchemeHandler` (iOS) and `WebView.addJavascriptInterface`/`evaluateJavascript` + `WebViewClient.shouldInterceptRequest` (Android) with the SAME seam semantics — so `explore-web3-capabilities`'s provider + `ipfs://` work is backend-agnostic across desktop AND mobile. Note any semantic gap (e.g. main-thread/coroutine constraints, security-origin/`isSecure` scheme traits differing per platform) as a finding.
6. **Headless / automatable verification on mobile.** The desktop webview proofs run under Xvfb in dedicated `shell-*` steps kept out of the display-free `zig build test` gate. What is the mobile analogue — simulator/emulator boot + a scripted navigate/snapshot assertion (iOS Simulator + `xcrun simctl`, Android emulator + `adb`/instrumented test), run in a dedicated build/CI leg — that proves the seam end-to-end without a physical device, while keeping the core gate device-free? Decide the strategy; do NOT fold mobile proofs into `zig build test`.

## Problem Statement

wezig is being built as "a browser as a browser should be", and a browser people actually use has to be where they are — which today means iOS and Android as much as the desktop. The webview-shell track (ADR-0005/0007) is how wezig becomes usable and differentiated long before the native renderer matures, and its whole architectural bet is TWO pinned, swappable seams: the `Renderer` content backend and the `Toolkit` chrome host, with a chrome that talks only to those seams and reaches neither `webkit_` nor `gtk_` (machine-checked by `src/chrome_conformance.zig`). That design PROMISES mobile is a backend swap — a `WKWebView`/Android `WebView` `Renderer` and a UIKit/Android `Toolkit`, with the chrome and the web3 hooks unchanged. But nothing in the repo has proven that promise: the opaque `ViewHandle` is, in practice, a GtkWidget both current backends agree on (ADR-0007 explicitly calls cross-toolkit view embedding a "foreign-embedding spike, not a drop-in"); input/scroll forwarding is deferred at the `Renderer` seam; and Zig-into-an-`.ipa`/`.aab` (cross-compile, link, JNI/Obj-C interop, signing, packaging) is entirely unproven here. The mobile OSes also own the window and run loop, which the desktop `Toolkit` currently owns itself. Before committing to a mobile BUILD, we need CONFIDENCE that the seams really do carry mobile, the toolchain really works, and we know HOW the mobile browser grows — not a mobile app written on guesses.

## Solution

Explore, don't build the mobile app. Spike each risky part on the narrowest real case, behind the EXISTING seams wherever they hold, and feed any insufficiency back as a seam finding rather than working around it:

- **Toolchain spike (per platform).** Prove the `wezig` Zig library cross-compiles and links into a minimal native shell on each platform, and that the shell launches: iOS (static lib into an Xcode target, a `WKWebView`, code-signed to a simulator), Android (NDK build into an APK, JNI to an `android.webkit.WebView`, run on an emulator). Deliverable is a launching skeleton + a pinned, written toolchain path — not a real app.
- **Seam-attachment spike (per platform).** Implement a mobile `Renderer` backend (iOS `WKWebView`, Android `WebView`) satisfying the pinned `Renderer` interface, drive ONE real page through it (navigate + one lifecycle event observed + the view non-blank), and host that view through a mobile `Toolkit`/chrome-host — proving the opaque `ViewHandle` and the two seams carry mobile, OR pinning exactly where they don't.
- **Web3-hook spike (per platform).** Round-trip ONE bridge message both ways (`window.wezig.ping`) and serve ONE custom-scheme request (`wezig-test://hello`) through the mobile backend, mirroring the desktop `shell-bridge-test`/`shell-scheme-test` proofs, so the web3 capabilities are confirmed backend-agnostic across desktop AND mobile.
- **Verification strategy.** Decide and prove a simulator/emulator-based, scriptable proof leg (kept OUT of the display-free `zig build test` gate, exactly like the Xvfb `shell-*` steps), so the mobile seam can be regression-tested without a physical device.
- **Evaluate + decide the open questions** (target floor, toolchain pinning, `ViewHandle` sufficiency, chrome-host/lifecycle mapping, hook parity, verification), capture the load-bearing ones in an ADR, and emit a de-risked, SLICED mobile build plan (which follow-on build spec grows packaging / app lifecycle / full chrome / store delivery, in what order, to what bar).

Keep the existing seams (`Renderer` ADR-0005/0006, `Toolkit` ADR-0006) as the structure the mobile work extends; the chrome-binding-free discipline (`chrome_conformance.zig`) and the desktop `shell-*` proofs are the prior art the mobile proofs mirror.

## Implementation Decisions

Decisions made at launch to seed tasking (trimmed into tasks/ADRs at tasking-time):

- **Extend the existing seams; do not fork them.** Mobile `Renderer` backends implement the pinned `Renderer` VTable; the mobile chrome host implements (or minimally extends) the `Toolkit` seam. The chrome (`src/chrome.zig`) stays untouched and binding-free; the mobile backends are the ONLY code that touches platform webview/UI APIs, exactly as `system_webview_renderer.zig`/`gtk_toolkit.zig` are the only WebKitGTK/GTK touchers on desktop. Any need to change a pinned interface is a FINDING fed back to ADR-0006/0007, not a silent divergence.
- **`ViewHandle` stays opaque; prove it carries a mobile native view.** The iOS backend returns a `WKWebView`/`UIView` as the opaque handle; the Android backend a `WebView`/`View`. The spike either confirms the mobile `Toolkit` can embed it unchanged, or pins the exact contract the seam is missing.
- **Input/scroll forwarding remains deferred (as on desktop).** Mobile webviews self-handle input/scroll like the desktop system webview, so the seam's deferred input methods are not blocked by this spec; the exploration only NOTES that a future `WezigRenderer`-on-mobile would need them (consistent with ADR-0006's deferral), not adds them speculatively.
- **Verification mirrors the `shell-*` pattern, one tier down.** Dedicated build/CI legs boot a simulator/emulator, drive one navigation, and assert a lifecycle event + a non-blank snapshot — kept OUT of `zig build test`, exactly like the Xvfb `shell-test`/`shell-bridge-test`/`shell-scheme-test` steps. The mobile bridge/scheme proofs mirror `shell-bridge-test`/`shell-scheme-test` message-for-message.
- **A thin native shell project per platform is expected** (an Xcode target for iOS, a Gradle/NDK project for Android) hosting the Zig library, rather than driving the whole app from `build.zig`. The exact division of labour between `build.zig` cross-targets and the native project is a Q2 spike output.

## Testing Decisions

- **Seam contracts stay in `zig build test`, headless and device-free.** Any mobile-agnostic seam logic added is exercised through the existing `FakeRenderer`/`FakeToolkit` fakes in the display-free gate; no simulator/emulator is required for the core gate. The REAL mobile proofs (a page renders on `WKWebView`/Android `WebView`) live in the dedicated per-platform legs, mirroring how the WebKitGTK proof lives only in `shell-test` under Xvfb.
- **Test EXTERNAL behaviour, per the desktop prior art.** Assert what the desktop proofs assert — a page loads (a `.finished` lifecycle event reached a subscriber), the view is non-blank (a snapshot), the bridge round-trips both ways, the custom scheme serves a native body that renders — not platform-API internals. Prior art to copy: `renderer.zig`'s seam-contract tests (fakes, in-gate) and `shell_main.zig`'s `smoke`/`bridge`/`scheme` modes (real backend, under the wrapper).
- **One device surface per platform for the spike** (one iOS simulator, one Android emulator). Physical-device matrices, performance, and store validation are out of scope for the exploration.

## Out of Scope (this is exploration)

- **A shipped mobile browser** — packaging pipeline hardening, full mobile chrome (tabs, gestures, settings), app lifecycle/state restoration, permissions UX, notifications, store submission/signing for release. The exploration PROVES the seams + toolchain and PLANS the build; BUILDING the mobile app is a follow-on spec this de-risks.
- **The native renderer on mobile** (`WezigRenderer` running on iOS/Android) — this spec uses the SYSTEM webview backends only; the native-renderer-on-mobile question is downstream of both this spec and `explore-native-renderer`.
- **The web3 capabilities themselves** — this spec only confirms the two HOOKS carry on mobile webviews; the EIP-1193 provider and `ipfs://` resolution are `explore-web3-capabilities` and its follow-on builds.
- **The desktop shell and its seams** (`explore-webview-shell`, ADR-0005/0006/0007) — inherited as fixed points, not re-opened here (except to receive any seam-insufficiency finding this exploration surfaces).
- **Physical-device testing, cross-device performance, and the full OS-version support matrix** beyond the one floor chosen for the spike (Q1).

## Further Notes

- **Why a fourth exploration spec rather than folding mobile into `explore-webview-shell`.** `browser.md` was deliberately re-scoped (2026-07-14) to avoid one spec mixing a committed slice with gated direction, per "task a spec atomically or split it". Mobile has exactly the shape of the existing three explorations — open questions, unproven approach, needs a spike + a build plan, not a build — so it is authored as its own confidence-tier spec, ordered behind the seam it depends on. This keeps each spec atomically taskable.
- **What the seam design already buys mobile (the reason to be optimistic).** The two web3 hooks map near 1:1 to platform primitives (iOS `WKUserContentController` + `WKURLSchemeHandler`; Android `addJavascriptInterface`/`evaluateJavascript` + `WebViewClient.shouldInterceptRequest`), and the chrome is already machine-proven binding-free — so the differentiators and the chrome are positioned to be a backend swap. The exploration exists to convert "positioned to be" into "proven to be".
- **The sharpest risk to watch.** ADR-0007 already flagged cross-toolkit view embedding (a non-GTK chrome hosting a foreign view) as an unproven spike; mobile makes that a first-class, must-answer case (Q3/story 6) rather than a desktop edge case. If the opaque `ViewHandle` proves insufficient, that finding reshapes the pinned `Renderer`/`Toolkit` interfaces and is the most valuable single output of this exploration.
- **Landscape note (mid-2026 framing).** System-webview-backed cross-platform browsers are well-trodden (every non-Chromium mobile "browser" on iOS is a `WKWebView` host by App Store policy); wezig's differentiation on mobile is the SAME native Ethereum-provider + `ipfs://` story attached at the seam, not the webview hosting itself. That is why proving the two hooks on mobile (stories 8–9) is load-bearing for the whole mobile value proposition.
