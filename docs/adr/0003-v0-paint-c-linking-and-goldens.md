# v0 paint: SDL3 + stb_truetype linking, and the golden-image format

The v0 paint task (`paint-sdl3-stb-window`) lands the first real C bindings, so it fixes HOW C libraries are linked in this Zig project and HOW the paint output is regression-tested. These are hard to reverse (every future backend and every golden inherits them), so they are recorded here.

## The C-linking approach (Zig 0.16)

- **stb_truetype is VENDORED and compiled by Zig, linked into the LIBRARY module.** `src/vendor/stb_truetype.h` (public-domain, v1.26) plus a one-line implementation TU `src/vendor/stb_truetype_impl.c` (`#define STB_TRUETYPE_IMPLEMENTATION`) are compiled via `mod.addCSourceFile` with `mod.link_libc = true`. It is glyph rasterisation only (no window), so it belongs to the library module: the `StbSoftwareBackend` and the headless golden tests use it with NO SDL/window. Bindings are generated with `@cImport(@cInclude("stb_truetype.h"))` (no hand-written prototypes).
- **SDL3 is built FROM SOURCE via the `castholm/SDL` Zig-build-system port, linked into the APP EXECUTABLE only.** `build.zig.zon` pins `git+https://github.com/castholm/SDL.git` at a fixed commit+hash; `build.zig` links `sdl_dep.artifact("SDL3")` into the exe with `linkLibrary` (which also propagates SDL's headers so `@cInclude("SDL3/SDL.h")` resolves). SDL is NOT linked into the library module, so the test path never needs it. This is the load-bearing split: **SDL below the app entrypoint, stb below the paint seam, and the headless golden tests depend on neither a system SDL nor a display.**

Why build SDL from source rather than link a system `-lSDL3`: the `verify` gate + CI + subagents must build reproducibly on machines with no SDL dev package (this one has none, only a Steam-runtime copy invisible to pkg-config). The from-source port makes `zig build` self-contained. It unpacks bundled Linux system-deps into a regenerated `zig-pkg/` directory, which is git-ignored as a build artifact.

## The stb binding shape

`StbSoftwareBackend` holds one `c.stbtt_fontinfo` over a single embedded face and implements the `PaintBackend` vtable (ADR-0002): `measureRun` sums scaled `stbtt_GetCodepointHMetrics` advances (with `stbtt_GetCodepointKernAdvance`), `drawRun` walks the SAME advance loop and blends each `stbtt_GetCodepointBitmap` coverage bitmap as anti-aliased text, `fillRect`/`drawBorder` are software rectangle fills, and `beginFrame`/`present` are no-ops offscreen. Because measure and draw share one advance loop, measured width equals drawn width exactly. Shaping/measurement/raster all live below the seam; callers above it (`paintTree`) touch only the vtable.

## The embedded font

**Roboto Regular is VENDORED (`src/assets/Roboto-Regular.ttf`, Apache-2.0, license in `src/assets/Roboto-LICENSE.txt`) and `@embedFile`d.** Golden images depend on the EXACT font bytes + stb version, so the font must be pinned in-repo, not resolved from the host. v0 maps every `font-family`/`weight` to this one face (a documented v0 limit); `size_px` drives the stb scale.

## The golden-image format and tolerance

- **Format: PNG, 8-bit RGBA, non-interlaced, a single zlib STORED (uncompressed) block.** A minimal self-contained codec lives in `src/surface.zig` (encoder + a decoder that reads exactly what the encoder writes). Chosen over an external PNG/zlib dependency to keep the acceptance loop dependency-free; it is NOT a general PNG library.
- **Tests compare IN MEMORY against `@embedFile`d references**, so the test path is headless and touches no disk and no window. References live in `src/testdata/golden/` and are regenerated by `zig build gen-goldens` (a maintenance step, NOT part of `zig build test`, so an accidental raster change is caught rather than silently re-baselined).
- **Per-channel tolerance = 2.** Goldens are produced by THIS rasteriser on the pinned toolchain, so the surface is byte-identical in practice; the small tolerance absorbs only theoretical rounding noise. A real regression (a moved glyph, a wrong colour) shifts many pixels far past 2 and fails loudly.
- **Fixture set is deliberately SMALL** (a background+border box, one text line, and a combined "page fragment"), each a few tens of pixels, so the reference PNGs stay maintainable (the spec calls this out).

## Considered options

- **Link a system SDL3 (`-lSDL3` + pkg-config):** rejected â€” not reproducible; the gate/CI machine has no SDL dev package. The from-source port trades a slower first build for a self-contained one.
- **Raw `.rgba` golden buffers instead of PNG:** rejected â€” the spec names PNG references, and PNG stays inspectable by a human. The stored-block codec keeps that cheap.
- **Compress the golden PNGs (real deflate):** unnecessary for v0's tiny fixtures; STORED blocks avoid pulling in or writing a deflate encoder.

## Consequences

- SDL sits in `src/sdl.zig` + `src/main.zig` (the app entrypoint) ONLY; the library, the paint backend, and the golden tests are SDL-free, so the tests run in CI with no display.
- The future SDL+FreeType+HarfBuzz+Skia backend replaces `StbSoftwareBackend` behind the same vtable, and the golden tests (which target the offscreen surface) survive the swap, per ADR-0001.
- Refreshing goldens after an intentional raster change is `zig build gen-goldens`; the diff is reviewed like any other change.
