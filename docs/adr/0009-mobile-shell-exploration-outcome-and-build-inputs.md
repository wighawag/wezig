# The mobile-shell exploration outcome: the two pinned seams carry iOS + Android, and the build inputs it settled

The `explore-mobile-shell` exploration is complete. Its verdict: **yes, build the mobile browser on the SAME two pinned seams** ‚Äî the `Renderer` content seam (ADR-0005/0006) and the `Toolkit` chrome-host seam, now split into `ChromeSurface` + `HostLoop` (ADR-0008) ‚Äî because minimal iOS (`WKWebView`) and Android (`android.webkit.WebView`) backends drove a real page and both web3-load-bearing hooks (script-message bridge, custom-scheme interception) THROUGH those seams, and the opaque `ViewHandle` carried a mobile native view across a NON-GTK `ChromeSurface`‚Üî`Renderer` boundary on both platforms. The load-bearing decisions the six launch questions settled were CONFIRMED by the spikes (none was revised); the spikes surfaced per-platform semantic GAPS that are recorded here and fed to `explore-web3-capabilities` and the seam ADRs. The full findings + the sliced mobile build plan live in `docs/mobile-exploration-findings.md`; this ADR records only the load-bearing decisions and the spike outcomes, so the follow-on mobile BUILD spec inherits them as fixed points.

This ADR does not re-pin the interfaces (ADR-0006 pins the two seams; ADR-0008 pins the `Toolkit` split) and does not build the mobile app (the spec's Out-of-Scope). It records WHAT the exploration proved, the ONE seam refinement it fed back (the `Toolkit` split, already its own ADR-0008), and the per-platform gaps + build-plan choices it deliberately DEFERRED to the mobile build spec, so those are not silently re-litigated.

## Decisions this exploration settled (all CONFIRMED by the spikes ‚Äî none revised)

- **Target floor: iOS 16.0, Android API 26 (8.0).** iOS arm64 devices, spiked on the iOS **Simulator**; Android arm64-v8a + x86_64, spiked on an **x86_64 emulator**. The floors chosen for modern `WKWebView`/`WKURLSchemeHandler` and auto-updating-Chromium-WebView semantics held: the spikes never needed a legacy workaround. Runner ceiling recorded downstream: GitHub `macos-14` ships Xcode 15.4 ‚áí iOS SDK caps at 17, so the iOS proofs build+run against an **iOS 17 Simulator** while the **deployment-target floor stays 16.0**.

- **Toolchain: a Zig STATIC LIBRARY + a thin native shell per platform** (not driving the app from `build.zig`). Proven green in CI on both platforms:
  - **iOS** (`mobile-ios.yml`, `macos-14`): `zig build-lib -target aarch64-ios-simulator` (and `aarch64-ios` device, compile-proven) links into a minimal Swift/`WKWebView` app that launches on the simulator via `xcrun simctl` and calls the Zig C-ABI; **no code signing / no Apple Developer account** (Simulator only ‚Äî device/store signing is a follow-on build spec).
  - **Android** (`mobile-android.yml`, `ubuntu-latest`): the `wezig` static lib ‚Äî INCLUDING its `stb_truetype` C dep ‚Äî cross-links for both ABIs (`aarch64-linux-android.26`, `x86_64-linux-android.26`) against the **NDK r26d** sysroot, packages into a debug APK behind a JNI shim. The single C-libc gap (bionic `<math.h>` + arch headers) is closed by wiring the NDK sysroot into the C compile (`-Dmobile-sysroot=‚Ä¶`); the vendored stb C is compiled `-fno-sanitize=undefined` (mobile SDKs ship no static-link UBSan runtime); the mobile C-ABI exports are force-kept via a `root.zig` `comptime`.
  - Division of labour: **Zig owns the portable core + all seam logic; the OS toolchains own signing/packaging/entitlements.** The mobile backends reach the platform webview over a small C-ABI ops table (`WkPlatform` on iOS, `CJavaBridge`/JNI on Android) ‚Äî the mobile analogue of the desktop backend's inline `@cImport`, forced because the pinned toolchain has the native side own `WKWebView`/`android.webkit.*`.

- **The `Toolkit` split (`ChromeSurface` + `HostLoop`) ‚Äî the ONE seam refinement fed back, recorded in ADR-0008.** Mobile inverts lifecycle ownership: the OS owns the window + run loop (chrome lives inside a `UIViewController`/`Activity`), so the desktop `createWindow`/`present`/`run`/`quit` half cannot be honestly implemented. `Toolkit` was split along that axis into a `ChromeSurface` half (`embedView`, `setUrlText`, `setBackEnabled`/`setForwardEnabled`, `setChromeCallback` ‚Äî both platforms implement it) and a `HostLoop` half (desktop-only). `GtkToolkit` composes both; a mobile toolkit implements `ChromeSurface` only. `chrome.zig` is unchanged and the conformance guard stays green. This is the same discipline ADR-0006 used deferring input/scroll: refine a pinned interface on the second target's evidence.

- **`ViewHandle` sufficiency (Q3) ‚Äî CONFIRMED SUFFICIENT; the opaque contract stands, no refinement needed.** See the dedicated section below ‚Äî this was the exploration's single sharpest risk (ADR-0007 explicitly flagged cross-toolkit view embedding as a spike, not a drop-in).

- **The two web3 hooks carry on both mobile webviews ‚Äî CONFIRMED, with TWO documented per-platform gaps encoded as findings, NOT baked into the seam signature.** Both hooks round-trip through the SAME `Renderer`-seam methods as desktop (`injectUserScript`/`setScriptMessageHandler`/`evaluateScript`, `registerScheme`): iOS via `WKUserContentController` + `WKScriptMessageHandler` + `WKURLSchemeHandler`; Android via `addJavascriptInterface`/`evaluateJavascript` + `WebViewClient.shouldInterceptRequest`. The gaps (below) are the "scheme security traits at the seam" item ADR-0007 deferred, now forced by proving the hooks on mobile; they feed `explore-web3-capabilities`.

- **Verification: dedicated device-free legs, nightly/on-demand, OUT of `zig build test`** ‚Äî the mobile analogue of the desktop Xvfb `shell-*` steps. `mobile-verify.yml` (`workflow_dispatch` + nightly `schedule`, NEVER push/PR): iOS boots an iOS 17 Simulator via `xcrun simctl`, Android boots a headless KVM-accelerated x86_64 emulator, each asserting navigate + a `.finished` lifecycle event + a non-blank snapshot. The fast per-PR `mobile-ios.yml`/`mobile-android.yml` legs keep only the cheap BUILD proofs (cross-link + launch/APK); the expensive RUN proofs live ONLY in nightly `mobile-verify`. The core `zig build test` gate stays device-free (only the `Fake*` seam contracts run there).

## The Q3 result stated unambiguously: the opaque `ViewHandle` is SUFFICIENT across a non-GTK boundary

The `mobile-viewhandle-embedding-proof` spike (Q3/story 6) resolved ADR-0007's flagged cross-toolkit-embedding risk on the narrowest real case, on BOTH platforms. Outcome: **confirmed-sufficient ‚Äî the opaque `*anyopaque` `ViewHandle` carries a mobile native view across the mobile `ChromeSurface`‚Üî`Renderer` boundary with NO interface change.** ADR-0006's opaque handle and ADR-0008's split `Toolkit` are confirmed carriable to mobile; no typed-handle / handle-with-vtable refinement is required. Ground truth: `work/notes/findings/viewhandle-crosses-mobile-toolkit-boundary-2026-07-18.md`.

- **iOS (executed end-to-end):** the handle is the `WKWebView`'s `UIView*`; the Swift embed op downcasts it and `addSubview`s it. Asserted on the `ios-embedding-proof` CI leg by BOTH superview-identity (it was hosted THROUGH the seam) AND a non-blank `WKWebView.takeSnapshot`.
- **Android (the sharp risk):** the handle is a JNI **global-ref** to the `android.webkit.WebView`, not a raw pointer. It is carried by the SAME opaque contract because (a) **lifetime**: a `NewGlobalRef` survives past the producing JNI call and stays valid while the chrome holds the view ‚Äî the durability a raw pointer gives on desktop; the Zig `ChromeSurface.embedView` copies the bits through unchanged and only the native side interprets them; and (b) **thread-affinity**: the interpretation step (`ViewGroup.addView`) must run on the UI thread, but that is an Android view-hierarchy constraint on the NATIVE side, not a gap in the seam contract ‚Äî a typed handle would not change WHERE the embed must run. The live-emulator RUN of the instrumented embedding test is delegated to the KVM `mobile-verify` leg (`connectedAndroidTest`); the Q3 *contract* question is already answered YES by construction (the native link resolves), the headless Zig seam-contract tests, and the JNI global-ref semantics ‚Äî the emulator run is a regression check, not an open contract question.

**Therefore ADR-0006's opaque `ViewHandle` stands.** The chrome-surface half stays backend-agnostic (it embeds an opaque handle; only the backends interpret it), preserving the seam discipline across a non-GTK toolkit‚Üîbackend boundary.

## Per-platform findings the spikes surfaced (recorded, fed downstream)

Each is a verified finding under `work/notes/findings/`; the mobile build spec + `explore-web3-capabilities` inherit them:

- **iOS `WKURLSchemeHandler` is lifecycle-ordered relative to view creation.** A custom scheme handler MUST be installed on the `WKWebViewConfiguration` BEFORE the `WKWebView` is created (the webview copies its config at `init`; a handler set afterward is ignored). Desktop WebKitGTK and Android's per-`WebViewClient` hook have no such ordering. The seam's `registerScheme` shape is unchanged; the constraint is surfaced at the iOS backend + recorded. Implication: the mobile chrome (and the N-`PageContext` model, ADR-0007) must thread the scheme set into EACH webview's config at build time. (`ios-wkurlschemehandler-registration-ordering-2026-07-18.md`)
- **Android callbacks run on non-UI (binder) threads ‚Äî and the two hooks have OPPOSITE thread contracts.** `WebViewClient` load callbacks and the bridge post are marshalled onto the UI thread (via a main-`Looper` `Handler`) BEFORE crossing the seam, so the chrome sees lifecycle events exactly as on desktop (iOS's `WKNavigationDelegate` already fires on the main queue ‚Äî no equivalent gap). BUT `shouldInterceptRequest` must answer SYNCHRONOUSLY on the binder thread (the WebView blocks) ‚Äî it CANNOT hop to the UI thread ‚Äî so the custom-scheme up-call is the ONE seam callback that legitimately runs off the host/UI thread, and the seam `SchemeHandler` it invokes must be thread-safe and touch no UI state. (`android-webviewclient-nonui-thread-marshalling-2026-07-18.md`, `android-custom-scheme-nonui-thread-and-opaque-origin-2026-07-18.md`)
- **Android custom schemes are OPAQUE / insecure origins by default ‚Äî the `ipfs://` gap.** A request served via `shouldInterceptRequest` for a custom scheme is an opaque origin: `isSecureContext` is false, `crypto.subtle` is unavailable, service workers cannot register, CORS differs. There is NO public `WebView` API to promote a custom scheme to secure/CORS-enabled (unlike WebKitGTK's `WebKitSecurityManager.register_uri_scheme_as_secure`); the closest lever is serving under a synthetic secure `https://appassets.androidplatform.net` origin (`WebViewAssetLoader`). So `ipfs://` behaviour is **NOT backend-identical** for secure-context/SW-dependent pages: the mobile build + `explore-web3-capabilities` must decide either (a) serve `ipfs://` under a synthetic secure origin, or (b) accept a non-secure origin on Android. This IS the "scheme security traits at the seam" decision ADR-0007 deferred, now forced. (`android-custom-scheme-nonui-thread-and-opaque-origin-2026-07-18.md`)
- **Android `injectUserScript` is `evaluateJavascript`, not document-start injection.** Android's WebView has no equivalent to iOS's `WKUserScript(.atDocumentStart)` / WebKitGTK's user-content-manager document-start injection; a real provider bridge must re-inject on each page start (e.g. an `onPageStarted` hook). Recorded for `explore-web3-capabilities`'s provider injection on Android. (`mobile-web3-hooks-parity-decisions-2026-07-18.md`)

## Cross-references to the sibling explorations

- **`explore-web3-capabilities` inherits a cross-platform-validated hook contract.** The two hooks are confirmed backend-agnostic across desktop AND both mobile webviews through the SAME `Renderer`-seam methods, so its EIP-1193 provider + `ipfs://` work is written once. It also inherits the two mobile gaps above (Android non-secure-origin `ipfs://`; Android non-document-start injection) plus the desktop wallet-boundary decision (ADR-0007), as explicit inputs to its security model.
- **`explore-native-renderer` inherits the mobile-webview-only scope.** This exploration uses the SYSTEM webview backends only; running `WezigRenderer` on iOS/Android is downstream of BOTH this spec and `explore-native-renderer` and is deliberately out of scope here. When it lands, `WezigRenderer` on mobile satisfies the SAME `Renderer` seam these mobile backends do, and must reproduce the two hooks' semantics (including the scheme-security traits) to stay backend-identical.

## Consequences

- The follow-on mobile BUILD spec is authorable from `docs/mobile-exploration-findings.md` alone, with the seam + toolchain proven; its slices each build against a seam whose surface and deferred decisions are recorded. Packaging pipeline, app lifecycle/state restoration, full mobile chrome (tabs/gestures/settings), permissions, and code-signing + store delivery are placed THERE (out of this exploration).
- The two-seams pinning (ADR-0006) and the `Toolkit` split (ADR-0008) stand; this ADR neither supersedes nor amends them ‚Äî it records the mobile exploration's outcome and the build inputs that reference them.
- One carry-forward HAZARD the build spec must fix (not a spike blocker): the Android renderer creates a NEW JNI global-ref on every `Renderer.view()` call and never deletes it, and the embedding shim leaks its `EmbedCtx` global-ref at teardown ‚Äî fine for a one-shot spike, a per-embed ref LEAK the build must fix (cache one global-ref per view, delete on teardown). (`viewhandle-crosses-mobile-toolkit-boundary-2026-07-18.md`)
