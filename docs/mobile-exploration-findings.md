# Mobile-shell exploration: findings + de-risked, sliced build plan

This is the CONFIDENCE deliverable of the `explore-mobile-shell` exploration (stories 11, 12, 13; ADR-0009): a durable, grounded report of what the eight mobile spikes PROVED about running wezig on iOS and Android through the two pinned seams, plus a de-risked, SLICED build plan for the follow-on mobile-browser BUILD spec(s). An exploration's "done" is confidence + a plan, not a shipped mobile app (spec `explore-mobile-shell`), so this document — not the spike code — is what a human or agent reads to author that build spec with the mobile browser's real shape already known.

It is the mobile analogue of `docs/shell-exploration-findings.md` (which fed the desktop build plan). The load-bearing DECISIONS the exploration settled are pinned in **ADR-0009**; the ONE seam refinement it fed back (the `Toolkit` split) is **ADR-0008**; the two pinned seams stay **ADR-0006**. Every claim below traces to something a spike actually observed or a verified finding under `work/notes/findings/`; those are cited inline.

The eight spikes, in dependency order, were:

- `mobile-toolkit-seam-split` (story 7): split the `Toolkit` seam into a `ChromeSurface` half (both platforms implement) and a `HostLoop` half (desktop-only), so the OS-owned-run-loop inversion on mobile is honest. Landed as **ADR-0008**.
- `android-toolchain-ndk-crosslink` (stories 1, 3) + `ios-toolchain-crosslink` (stories 1, 2): the Zig-static-lib + thin-native-shell toolchain, per platform, proven green in CI.
- `android-renderer-backend-oneshot` (story 5) + `ios-renderer-backend-oneshot` (story 4): a mobile `Renderer` backend per platform driving one real page through the pinned seam.
- `mobile-viewhandle-embedding-proof` (story 6): the opaque `ViewHandle` crossing the mobile `ChromeSurface`↔`Renderer` boundary — the exploration's sharpest risk.
- `mobile-web3-hooks-parity` (stories 8, 9): the two web3 hooks on both mobile webviews.
- `mobile-verification-legs-ci` (story 10): the dedicated device-free verification legs.

## 1. What landed, and where the seam held

The exploration extended the SAME two seams the desktop shell pinned, plus the `Toolkit` split (ADR-0008), with no new seam invented:

- **`Renderer` seam** (`src/renderer.zig`) gained two mobile implementations: `IosWebviewRenderer` (`src/ios_webview_renderer.zig`, over `WKWebView`) and the Android backend (`src/android_renderer.zig`, over `android.webkit.WebView`). Each satisfies the pinned VTable (navigate/reload/stop, back/forward, `view() -> ViewHandle`, `setViewportSize`, `setLifecycleCallback` → the `LifecycleEvent` union) and the two web3 hooks. Unlike the desktop backend — which `@cImport`s WebKitGTK and calls its C symbols inline — the mobile backends reach the platform webview over a small C-ABI **ops table** (`WkPlatform` on iOS, `CJavaBridge`/JNI on Android): the pinned toolchain has the native side (Swift / Java) own the webview, so the physical webview calls sit behind the C-ABI the toolchain pinned, while the seam impl stays pure Zig (so its contract tests run headlessly in `zig build test`). (Decisions: `ios-renderer-backend-c-abi-ops-table-decision-2026-07-18.md`.)
- **`Toolkit` seam** (`src/toolkit.zig`) was split (ADR-0008) into `ChromeSurface` (`embedView`, `setUrlText`, `setBackEnabled`/`setForwardEnabled`, `setChromeCallback`) and `HostLoop` (desktop-only `createWindow`/`present`/`run`/`quit`). `GtkToolkit` composes both; the mobile `MobileChromeSurface` (`src/mobile_chrome_surface.zig`) implements `ChromeSurface` ONLY and drives the native view hierarchy over an `EmbedPlatform` ops table (Swift `addSubview` / JNI `ViewGroup.addView`). `src/chrome.zig` is unchanged; the `chrome_conformance` guard (no `gtk_`/`webkit_` in the chrome) stays green. (Decisions: `mobile-chrome-surface-embed-ops-table-decision-2026-07-18.md`.)

**The seam surface was sufficient for mobile.** The mobile spikes drove one real page (navigate + a `.finished` lifecycle event + a non-blank snapshot), embedded the renderer's view through the chrome-surface, and round-tripped both web3 hooks — entirely through these methods, with no gap that blocked a spike and NO seam-signature change. The one seam refinement (the `Toolkit` split) was fed back BEFORE any mobile backend was written, exactly so a mobile toolkit never has to stub `createWindow`/`run`.

## 2. The Q3 result: the opaque `ViewHandle` carries mobile — CONFIRMED SUFFICIENT

This was the exploration's single sharpest risk. ADR-0007 explicitly flagged cross-toolkit view embedding (a non-GTK chrome hosting a foreign view) as "a foreign-embedding spike, not a drop-in"; mobile makes it a first-class must-answer case. **Outcome: confirmed-sufficient — the opaque `*anyopaque` `ViewHandle` carries a mobile native view across the mobile `ChromeSurface`↔`Renderer` boundary on both platforms, with NO interface change.** No typed-handle / handle-with-vtable refinement is required; ADR-0006's opaque handle stands. (Ground truth: `viewhandle-crosses-mobile-toolkit-boundary-2026-07-18.md`.)

- **iOS (executed end-to-end):** the handle is the `WKWebView`'s `UIView*`; the Swift embed op downcasts and `addSubview`s it. The `ios-embedding-proof` leg asserts BOTH superview-identity (hosted THROUGH the seam) AND a non-blank `takeSnapshot`.
- **Android (the sharp risk):** the handle is a JNI **global-ref** to the `WebView`, not a raw pointer, and it is carried by the SAME opaque contract:
  - **Lifetime** — a `NewGlobalRef` survives past the producing JNI call and stays valid while the chrome holds the view (the durability a raw pointer gives on desktop). The Zig `ChromeSurface.embedView` copies the bits through UNCHANGED; only the native side interprets them.
  - **Thread-affinity** — the interpretation step (`ViewGroup.addView`) must run on the UI thread, but that is an Android view-hierarchy constraint on the NATIVE side, not a gap in the seam contract. A typed handle would not change WHERE the embed must run.

  The live-emulator RUN of the instrumented embedding test is delegated to the KVM `mobile-verify` leg (`connectedAndroidTest`); the Q3 CONTRACT question is already answered YES by construction + the headless Zig seam-contract tests + the JNI global-ref semantics — the emulator run is a regression check, not an open question.

**Why this matters for the build:** the mobile chrome hosts the renderer's view through `ChromeSurface` with no per-platform view-type leak into the chrome; the only platform code is the native embed op, symmetric with the desktop `GtkToolkit.embedView`. **One native-side rule to carry forward** (documented, not a seam change): perform the embed on the UI thread — this composes with the Android thread-marshalling finding below.

## 3. The two web3 hooks on mobile: parity confirmed, two gaps recorded

Both hooks carry through BOTH mobile backends via the SAME seam methods as desktop (`injectUserScript`/`setScriptMessageHandler`/`evaluateScript`, `registerScheme`), mirroring the desktop `shell-bridge-test`/`shell-scheme-test`. iOS: `WKUserContentController` + `WKScriptMessageHandler` + `evaluateJavaScript`; `WKURLSchemeHandler`. Android: `addJavascriptInterface`/`evaluateJavascript`; `WebViewClient.shouldInterceptRequest`. The seam signatures are UNCHANGED — the per-platform gaps are surfaced at the backends + as findings, NOT baked into the seam (that trait API is `explore-web3-capabilities`'s call, per ADR-0007). (Decisions: `mobile-web3-hooks-parity-decisions-2026-07-18.md`.)

The four gaps a downstream reader must NOT be surprised by:

1. **iOS scheme registration is lifecycle-ordered.** `WKURLSchemeHandler` MUST be installed on the `WKWebViewConfiguration` BEFORE the `WKWebView` is created (the webview copies its config at `init`). Desktop and Android have no such ordering. Implication: the mobile chrome (and the N-`PageContext` model, ADR-0007) must thread the scheme set into EACH webview's config at build time — it cannot lazily add a scheme after a view exists. (`ios-wkurlschemehandler-registration-ordering-2026-07-18.md`)
2. **Android callbacks run on non-UI (binder) threads, and the two hooks have OPPOSITE contracts.** Load callbacks + the bridge post are marshalled onto the UI thread before crossing the seam (async reply), so the chrome sees events exactly as on desktop; iOS's `WKNavigationDelegate` already fires on the main queue. BUT `shouldInterceptRequest` must answer SYNCHRONOUSLY on the binder thread — the WebView blocks — so it CANNOT hop threads: the custom-scheme up-call is the ONE seam callback that runs off the UI thread, and the seam `SchemeHandler` must be thread-safe and touch no UI state. (`android-webviewclient-nonui-thread-marshalling-2026-07-18.md`, `android-custom-scheme-nonui-thread-and-opaque-origin-2026-07-18.md`)
3. **Android custom schemes are OPAQUE / insecure origins — the `ipfs://` gap.** Content served via `shouldInterceptRequest` for a custom scheme is an opaque origin: `isSecureContext` false, `crypto.subtle` unavailable, no service workers, CORS differs. There is NO public `WebView` API to promote a custom scheme to secure (unlike WebKitGTK's `WebKitSecurityManager`); the closest lever is serving under a synthetic secure `https://appassets.androidplatform.net` origin (`WebViewAssetLoader`). So `ipfs://` behaviour is **NOT backend-identical** for secure-context/SW-dependent pages — a real decision the build + `explore-web3-capabilities` must make (secure synthetic origin vs. accept non-secure on Android). This IS the "scheme security traits at the seam" decision ADR-0007 deferred, now forced. (`android-custom-scheme-nonui-thread-and-opaque-origin-2026-07-18.md`)
4. **Android `injectUserScript` = `evaluateJavascript`, not document-start injection.** Android has no equivalent to iOS's `WKUserScript(.atDocumentStart)` / WebKitGTK's document-start injection; a real provider bridge must re-inject on each page start (e.g. an `onPageStarted` hook). (`mobile-web3-hooks-parity-decisions-2026-07-18.md`)

## 4. Toolchain, pinned (what the build inherits, no re-derivation)

- **iOS** (`mobile-ios.yml`, `macos-14`, Xcode 15.4): `zig build-lib -target aarch64-ios-simulator` (exercised) + `aarch64-ios` (device, compile-proven) → a minimal Swift/`WKWebView` `.app` (hand-assembled via `mobile/ios/build-and-run.sh`, no committed `.xcodeproj` to drift), launched on an iOS 17 Simulator via `xcrun simctl`. Deployment-target floor **iOS 16.0**; runner SDK ceiling 17. Link details: `swiftc -parse-as-library`, `-Xlinker -force_load` the Zig archive, stb compiled `-fno-sanitize=undefined`, exports force-kept via `root.zig` comptime. **No signing / no Apple Developer account (Simulator only).**
- **Android** (`mobile-android.yml`, `ubuntu-latest`): **NDK r26d**; `zig build android-lib` cross-links the wezig static lib + its `stb_truetype` C dep for `aarch64-linux-android.26` + `x86_64-linux-android.26` against the NDK sysroot (C-libc gap closed via `-Dmobile-sysroot=…`), packaged into a debug APK behind a JNI shim (AGP 8.5.2 / Gradle 8.7 / minSdk 26 / compileSdk 34; ABIs arm64-v8a + x86_64). Pinned facts also in `mobile/android/README.md` / `mobile/ios/README.md`.
- **Division of labour:** Zig owns the portable core + all seam logic; the OS toolchains own signing/packaging/entitlements. This is the load-bearing constraint the build spec's packaging pipeline extends.

## 5. Verification strategy (the mobile analogue of the desktop Xvfb legs)

`mobile-verify.yml` — ONE dedicated workflow (`workflow_dispatch` + nightly `schedule`, NEVER push/PR) — is the mobile analogue of the desktop `shell-*` steps in `ci.yml`, one tier down:

- **iOS leg** (`macos-14`): boot iOS 17 Simulator via `simctl`, install + launch the `WKWebView` proof, assert navigate + `.finished` seam event + non-blank `takeSnapshot`.
- **Android leg** (`ubuntu-latest`, KVM): enable KVM → boot a headless x86_64 emulator (API 26, `-no-window`) via `reactivecircus/android-emulator-runner@v2` → `connectedDebugAndroidTest` (the instrumented `RendererSeamTest`: navigate + `.finished` seam event + non-blank bitmap).

The cheap BUILD proofs (cross-link + launch/APK) stay on the fast per-PR `mobile-ios.yml`/`mobile-android.yml` legs; the expensive RUN proofs live ONLY in nightly `mobile-verify` (iOS uses macOS runner minutes — free+unlimited on this public repo, but slower). **The core `zig build test` gate stays device-free** — only the `Fake*` seam contracts run there. This is the "dedicated leg" discipline ADR-0007 set for the desktop webview proofs, reused for mobile. (Decisions: `mobile-verify-legs-decisions-2026-07-18.md`.)

## 6. The de-risked, SLICED mobile BUILD PLAN

This section states the follow-on mobile BUILD spec(s) — their scope, ordering, and the bar each must hit — so they can be authored and tasked atomically from THIS document alone (a follow-on `to-spec` could work from it), mirroring how `docs/shell-exploration-findings.md` fed the desktop build. Everything below is grounded in a spike finding above; the "decide" points are the load-bearing choices the exploration surfaced but (correctly, being out of scope) did NOT settle.

### What the build spec already knows (de-risked by the exploration)

- The two seams (`Renderer` + the split `Toolkit`) carry mobile; the opaque `ViewHandle` crosses a non-GTK boundary (§2). The build builds AGAINST them, never around them.
- The Zig-static-lib + thin-native-shell toolchain is proven green in CI on both platforms (§4); packaging extends it, does not re-derive it.
- The two web3 hooks carry on both mobile webviews (§3), so `explore-web3-capabilities`'s provider + `ipfs://` work is backend-agnostic — with the two per-platform gaps (iOS scheme ordering; Android non-secure `ipfs://` origin + non-document-start injection) as known inputs.
- The device-free verification strategy (nightly `mobile-verify`, core gate device-free) is established and reusable per new mobile feature (§5).

### Slicing (recommended follow-on specs, in order)

The mobile build is bigger than one atomically-taskable spec; it slices into (at least) three follow-on build specs, ordered by dependency. A `build-mobile-shell` spec is the foundation; `build-mobile-chrome` and `deliver-mobile-signing-and-store` follow it. Web3-on-mobile is NOT a mobile-build slice — it is `explore-web3-capabilities`'s follow-on, inheriting §3.

**Slice A — `build-mobile-shell` (foundation; do FIRST).** Turn the per-platform spike shells into ONE real app skeleton per platform hosting the mobile `Renderer` + `ChromeSurface` on the settled seams. Scope: the packaging pipeline (a maintainable iOS project + Android Gradle project that build the Zig lib as a normal step, replacing the hand-assembled spike scripts), app lifecycle + state restoration (background/foreground, `WKWebView`/`WebView` state save-restore, the `UIViewController`/`Activity` host wiring), and a single-page chrome with a URL field + back/forward driven through `ChromeSurface`. **Bar:** the app builds via the real project (not the spike scripts), launches on device + simulator/emulator, drives one page through the seam with the `mobile-verify` assertions green, survives a background/foreground round-trip without losing page state, and FIXES the per-embed JNI global-ref LEAK the spike flagged (cache one ref per view, delete on teardown — §Consequences of ADR-0009). Must DECIDE: how app lifecycle maps to the seam (does `Renderer` gain suspend/resume, or is it host-only?); the Android `injectUserScript` re-injection point (`onPageStarted`).

**Slice B — `build-mobile-chrome` (full chrome; after A).** The full mobile chrome: tabs (over the N-`PageContext` model ADR-0007 designed — NOT baked one-visible-page-at-a-time), gestures (back-swipe, pull-to-refresh), settings, and permissions UX (camera/mic/location/notifications prompts the OS mediates and the webview surfaces). **Bar:** N `PageContext`s presented through `ChromeSurface.embedView(selected.view())` with background contexts still running; gestures + settings usable on device; each OS permission prompt handled end-to-end. Must DECIDE: the mobile presentation model's relationship to the desktop N-context design (shared or platform-specific chrome); which permissions ship v1.

**Slice C — `deliver-mobile-signing-and-store` (delivery; LAST, explicitly OUT of this exploration).** Code signing + store delivery: iOS device signing (Apple Developer account, provisioning profiles, entitlements — the Simulator needed NONE of this, so it is entirely un-spiked and lives HERE), Android release signing (keystore, Play App Signing), the `.ipa`/`.aab` release pipeline, and App Store / Play Store submission (review requirements, privacy manifests, the App-Store-policy constraint that non-`WKWebView` engines are forbidden — a landscape input, not a wezig blocker since mobile IS a `WKWebView` host). **Bar:** a signed release artifact per platform, accepted by each store's validation. This slice deliberately assumes A + B are done; it is pure delivery, and it is the clearest example of what the exploration placed OUT of scope.

### What every mobile build slice must DECIDE (the load-bearing choices the exploration surfaced)

- **`ipfs://` secure-context posture on Android (highest priority for web3-on-mobile; §3.3).** Android custom schemes are non-secure opaque origins with no promotion API. DECIDE, WITH `explore-web3-capabilities`: serve `ipfs://` under a synthetic secure `https://` origin (app-assets style, to get a secure context + SW support) vs. accept a non-secure origin on Android. This is the "scheme security traits at the seam" decision ADR-0007 deferred; whatever is decided, the seam may grow the trait fields there (not here).
- **iOS scheme-set-at-config-build-time threading (§3.1).** The mobile chrome's per-view construction must install the whole scheme set on each `WKWebViewConfiguration` before the view exists; the N-`PageContext` model must thread it into every context. Decide where the scheme registry lives so each new webview config gets it.
- **App-lifecycle mapping to the seam (Slice A).** Does the `Renderer` seam gain suspend/resume/state-restore methods (so a `WezigRenderer` reproduces them), or is lifecycle a host-only concern above the seam? Decide before the second mobile feature depends on it.
- **Mobile presentation model vs. the desktop N-`PageContext` design (Slice B).** ADR-0007 designed N contexts decoupled from presentation for desktop; decide whether mobile tabs share that model or diverge, and expose session/partition identity the same way.
- **The per-embed JNI global-ref lifecycle (Slice A; carry-forward hazard).** The spike leaks a global-ref per `view()` call; the build MUST cache one ref per view and delete on teardown.

Authored this way, each mobile build slice (shell foundation, full chrome, signing/delivery) is independently taskable against a seam + toolchain whose real shape and gaps are now known — the "authorable-into-a-spec on its own" outcome the exploration exists to produce.

## Decisions recorded by this deliverable

This is a documentation task, but it makes judgement calls worth ratifying (recorded here and cross-linked from ADR-0009):

- **This findings doc lives at `docs/mobile-exploration-findings.md`, alongside `docs/shell-exploration-findings.md`,** NOT under `docs/adr/`. Rationale: it is a report + plan (a reference document), not a single decision; ADRs stay short (`work/protocol/ADR-FORMAT.md`), and the load-bearing DECISIONS are pinned in the companion ADR-0009. Alternative considered: fold everything into one long ADR; rejected because it violates the "an ADR can be a single paragraph" norm and buries the plan — the exact precedent `docs/shell-exploration-findings.md` + ADR-0007 set for the desktop exploration.
- **The mobile build is SLICED into three follow-on specs (§6), not one.** Rationale: the spec's Out-of-Scope explicitly separates packaging / full chrome / lifecycle / permissions / signing / store delivery, and "task a spec atomically or split it" forbids one spec mixing a foundation slice with a delivery slice. The slicing (`build-mobile-shell` → `build-mobile-chrome` → `deliver-mobile-signing-and-store`) is a PROPOSAL the human authoring the follow-on specs adopts and may re-cut; it is not a pinned interface. It touches no code and no other task — it is the plan's shape, flagged for ratification rather than silently assumed.
